# Page Deletion System Design

**Date:** 2026-03-30
**Status:** Draft
**Scope:** `packages/core/` and `scripts/` only — no SvelteKit routes, no public API. Admin/dev feature.

---

## Problem

The OP triplestore accumulates records for pages that no longer exist. When a URL returns 404 or another inaccessible response, its data should be cleanable — but only data *generated from that URL during indexing*, not data about it generated by other URLs.

---

## Constraints

- No public-facing API routes. Deletion is triggered via a standalone script or programmatically through the core library.
- Deletion must be scoped precisely: only triples where the dead URL is the **source** of the data.
- Both soft-delete (mark as unavailable) and hard-delete (remove triples) are supported.
- Error codes that trigger deletion are configurable.

---

## Data Model

### Soft Delete

Soft-deleting a page adds three predicates to its existing record. No other data is modified.

```
<url> octo:unavailable "404" .
<url> octo:unavailableSince 1743000000000 .
<url> octo:failCount 3 .
```

- `octo:unavailable` — the error code string that triggered the mark (e.g. `"404"`, `"410"`, `"timeout"`). Updated on each failure (always reflects the most recent error code).
- `octo:unavailableSince` — Unix timestamp of the **first** failure. Written only once; never overwritten on subsequent failures.
- `octo:failCount` — integer count of consecutive failures. Starts at 1 on first soft-delete; incremented on each subsequent failed check.

**`failCount` upsert pattern:** SPARQL has no atomic increment. `softDeletePage` must read the current `failCount` via `queryArray`, compute `n + 1`, then delete the old value and insert the new one using `query()` (not `insert()`, since `INSERT DATA` cannot contain a delete). The same delete-then-insert pattern applies to `octo:unavailable`. `octo:unavailableSince` uses a conditional write: it is only inserted if no value currently exists (checked via `queryBoolean`).

**Recovery:** if a soft-deleted URL returns 2xx on a later review, `clearSoftDelete(url)` removes the three predicates. Re-indexing is then triggered separately by the caller (the review script calls `client.indexSource(url)` after clearing). `clearSoftDelete` itself has no side effects beyond removing the markers.

### Hard Delete Scope

Hard delete removes only triples generated *from* the URL during its own indexing:

- All triples with `<url>` as subject (metadata, outgoing octothorpes, outgoing mentions)
- Blank nodes attached to `<url>`: requires a two-pattern DELETE clause (see SPARQL note below)
- `<origin> octo:hasPart <url>` — the parent–child link written during indexing
- **Origin special case:** if `<url>` is itself the registered origin (has `rdf:type octo:Origin`), also delete `octo:verified`, `octo:hasMember` triples, and the Origin type triple

**Blank node DELETE pattern:** `DELETE WHERE { <url> ?p ?o }` does not remove blank nodes that appear as *objects*. Blank nodes must be deleted with a two-part clause:

```sparql
DELETE {
  <url> octo:octothorpes ?bn .
  ?bn ?p ?o .
} WHERE {
  <url> octo:octothorpes ?bn .
  FILTER isBlank(?bn) .
  ?bn ?p ?o .
}
```

This is a separate update from the general subject-triple delete and must be run explicitly.

**Not deleted:**
- Inbound links from other pages: `<other> octo:octothorpes <url>` — generated by indexing `<other>`, not `<url>`
- Blank nodes from other pages that reference `<url>` as a target (their subject is `<other>`)
- Term usage timestamps: `<term> octo:used <timestamp>` — written on the term URI, not on the page being deleted

**Origin rule:** an Origin record is only cleaned up when the dead URL *is* the origin itself. If a subpage 404s, the Origin is untouched regardless of how many subpages remain.

## Architecture

### `packages/core/deleter.js` — `createDeleter` factory

Same dep-injection pattern as `createIndexer`. No SvelteKit imports.

```javascript
createDeleter({ query, queryArray, queryBoolean, instance })
```

Returns:

| Method | Description |
|--------|-------------|
| `softDeletePage(url, errorCode)` | Upserts `octo:unavailable` (always updated); writes `octo:unavailableSince` only on first call; increments `octo:failCount` via read-increment-upsert |
| `hardDeletePage(url)` | Removes all triples generated from `url`; applies blank node two-pattern delete; applies Origin special case |
| `clearSoftDelete(url)` | Removes `octo:unavailable`, `octo:unavailableSince`, `octo:failCount`. No side effects — caller is responsible for re-indexing if needed |
| `isSoftDeleted(url)` | Returns boolean — checks for presence of `octo:unavailable` |
| `getSoftDeletedPages()` | Returns `[{ url, errorCode, since, failCount }]` for all soft-deleted pages |
| `getAllPageUrls()` | Returns all `octo:Page` URIs (used to seed the review script) |
| `shouldHardDelete(record, config)` | Pure function — no SPARQL. Returns `true` if either threshold is met |

`shouldHardDelete(record, config)` accepts:
- `record: { since: number, failCount: number }`
- `config: { maxAgeDays: number, maxRetries: number }`

Returns `true` if `Date.now() - since > maxAgeDays * 86400000` **or** `failCount >= maxRetries`.

### `packages/core/indexer.js` — inline 404 detection

`createIndexer` gains two optional constructor deps: `deleter` (a `createDeleter` instance) and `deadCodes` (an array of HTTP status code strings/numbers, e.g. `[404, 410]`). Both default to `null`/`[]` — existing behaviour is unchanged when omitted.

When both are provided, `handler()` checks `subject.status` **before** calling `recordIndexing()`. This ordering is critical: a dead URL must not receive an updated `octo:indexed` timestamp. On a dead-code match:

1. Calls `deleter.softDeletePage(url, subject.status.toString())`
2. Throws an error to halt further processing (skipping `recordIndexing` and all subsequent steps)

`deadCodes` is fixed at construction time (factory dep), not passed per-call. This keeps `handler()`'s signature unchanged.

### `packages/core/index.js` — `createClient` integration

`createClient` wires `createDeleter` alongside `createIndexer` and passes `deadCodes` and the deleter instance into the indexer. `createClient` accepts an optional `deadCodes` array in its config (defaults to `[404, 410]`).

The full deleter instance is exposed on the returned client object as `client.deleter` (consistent with the existing `client.sparql` and `client.api` exposures). This gives callers access to `getAllPageUrls()`, `getSoftDeletedPages()`, `isSoftDeleted()`, etc. without needing to construct a separate deleter.

`createClient` also exposes a convenience method:

```javascript
deleteSource(url, { mode: 'soft' | 'hard', errorCode })
// Returns: { url, mode, deleted_at: number }
// Thin wrapper around deleter.softDeletePage / deleter.hardDeletePage
```

The review script accesses `client.deleter.getAllPageUrls()` and `client.deleter.getSoftDeletedPages()` directly. It does not need to import `createDeleter` independently.

`shouldHardDelete` is exported as a named module-level export from `deleter.js` (not a factory method), since it is a pure function with no instance state.

## Review Script (`scripts/review.js`)

Standalone Node.js script. Run via OS cron or manually. Uses `createClient` with config read from `.env` via `node --env-file=.env`.

### CLI Flags

| Flag | Default | Description |
|------|---------|-------------|
| `--dry-run` | false | Report only — no mutations |
| `--mode soft\|hard\|auto` | `auto` | Deletion mode. `auto` = soft first, promote on thresholds |
| `--dead-codes 404,410` | `404,410` | Comma-separated HTTP status codes (and `timeout`) to treat as dead |
| `--max-age-days N` | `30` | Days before soft-delete promotes to hard (auto mode) |
| `--max-retries N` | `3` | Fail count before soft-delete promotes to hard (auto mode) |
| `--concurrency N` | `5` | Parallel URL checks |

### Flow

1. Fetch all `octo:Page` URLs via `deleter.getAllPageUrls()`
2. For each URL (batched per `--concurrency`):
   a. If already soft-deleted: run `shouldHardDelete()`. If threshold met, hard-delete and continue (no fetch needed)
   b. Fetch the URL with a reasonable timeout (5s)
   c. If status matches `deadCodes` (or fetch throws a network error): call `softDeletePage()` or `hardDeletePage()` per `--mode`; network errors use error code `"timeout"`
   d. If status 2xx and URL was previously soft-deleted: call `clearSoftDelete()`, then `client.indexSource(url)` to re-index
3. Print structured report

### Report Format

```
Review complete: 142 URLs checked
  Dead (newly flagged):    3  [404: example.com/old-post, ...]
  Hard-deleted:            1  [gone.example.com]
  Recovered:               0
  Errors (fetch failed):   2

Dry-run mode: no changes were written.
```

## Out of Scope

- Public-facing deletion API routes
- User-initiated deletion (no accounts/auth in OP Core)
- Deleting Terms or Origins independently (only as a side effect of hard-deleting a page that is itself an Origin)
- Cascading deletion of inbound links from other pages
